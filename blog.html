7 Best WordPress Security Plugins to Install Before Launch (2026 Guide)

A breakdown of the exact security stack I use on every WordPress build — and why each plugin earns its place.

After building and maintaining 100+ WordPress sites, I've narrowed my standard security stack down to seven plugins. These aren't suggestions — they ship on every single build I deliver. Here's why each one made the cut.

1. Inactive Logout

Automatically logs out users after a set period of inactivity. If a client steps away from their dashboard on a shared computer, this prevents unauthorized access. Set it to 15 minutes — clients rarely notice, but it closes a real attack vector.

2. WP Activity Log

Tracks every change made in the WordPress dashboard — who logged in, what was edited, when settings were changed. When a client says "I didn't touch anything" and the site breaks, this log tells you exactly what happened. It's also critical for compliance-sensitive businesses.

3. UpdraftPlus

Automated backups to remote storage (Google Drive, Dropbox, or S3). I schedule daily database backups and weekly full-site backups. The one-click restore has saved client sites more times than I can count — especially after failed plugin updates.

4. WP 2FA (Two-Factor Authentication)

Adds TOTP-based two-factor authentication to all admin and editor accounts. This single plugin blocks the majority of brute-force login attacks. I make it mandatory for every user with publishing privileges.

5. Admin Login URL Change

Moves /wp-admin and /wp-login.php to a custom URL. This isn't security through obscurity alone — it stops automated bots from even finding the login page, which dramatically reduces server load from brute-force attempts.

6. Limit Login Attempts Reloaded

Rate-limits login attempts by IP and username. After 4 failed attempts, the IP gets locked out for 20 minutes. After 3 lockouts, it's blocked for 24 hours. This prevents sustained brute-force attacks without affecting legitimate users.

7. AB Split Test

Not a security plugin in the traditional sense — but I include it because it helps clients test changes safely before pushing them live. Rather than making irreversible changes to a production page, they can split-test variations. It's security for your conversion rate.

My rule: If a WordPress site leaves my hands without these seven plugins active and configured, it's not done. This is the baseline — additional security layers are added based on the project's risk profile.

WordPress Multisite vs Single Site: Which Setup Is Best for Business?

If you run multiple locations or brands, this decision affects your budget, performance, and sanity long-term.

When clients with multiple locations ask about their next website, this is the first question I walk through with them. The wrong choice here costs thousands to undo later.

When Multisite Makes Sense

  • Shared branding: All locations use the same theme, same look, same plugins. You update once, it rolls out everywhere.
  • Centralized management: One dashboard for all sites. One set of updates. One hosting plan. This is ideal for franchise operations.
  • Cost efficiency: You pay for one hosting environment instead of multiple. Plugin licenses are typically shared across the network.

When Separate Installs Win

  • Different designs per location: If each branch needs its own branding, layout, or content strategy, Multisite becomes a constraint, not a convenience.
  • Independent teams: If each location has its own marketing person making changes, separate installs prevent one admin from accidentally affecting another site.
  • Risk isolation: If one site gets compromised or breaks, the others are unaffected. With Multisite, a bad plugin update can take down the entire network.

Real Example: Franchise vs. Independent

I built a multi-location franchise site where every location needed the same look but different content. Multisite was the clear winner — one theme update, one plugin set, 12 locations all in sync.

On the flip side, I worked with a client running three completely different brands. Same owner, different industries. Multisite would've been a nightmare — separate installs gave each brand its own identity.

My recommendation: If the sites share 80%+ of their DNA (theme, plugins, structure), go Multisite. If they share less than 50%, go separate. In the gray zone, lean toward separate — it's easier to merge later than to split apart.

How to Fix a Crashed WordPress Site Without Losing Data (Step-by-Step)

Step-by-step walkthrough of how I've recovered sites from fatal errors, failed updates, and broken restores.

Your WordPress site is showing a white screen of death, a critical error message, or a 500 internal server error. Here's exactly what I do — in order — when a client calls in with a crashed site.

Step 1: Don't Panic. Access Your Hosting

Go to cPanel or your hosting dashboard. Not WordPress. If WordPress is crashed, the admin won't load. You need file-level or database-level access.

Step 2: Enable WordPress Debug Mode

Open wp-config.php via File Manager or FTP. Set WP_DEBUG to true and WP_DEBUG_LOG to true. Reload the site and check /wp-content/debug.log — this tells you exactly which file and line caused the crash.

Step 3: Identify the Cause

In 90% of cases, it's one of these:

  • A plugin conflict — usually after an update. The debug log will reference the plugin's folder name.
  • A theme issue — broken template file or deprecated function call.
  • PHP version mismatch — your hosting upgraded PHP and a plugin isn't compatible.
  • Memory exhaustion — the error log says "Allowed memory size exhausted."

Step 4: Fix Without the Dashboard

  • Plugin conflict: Rename the plugin's folder via FTP (e.g., /plugins/broken-plugin → /plugins/broken-plugin-disabled). Site should come back. Then troubleshoot the plugin separately.
  • Theme issue: Rename the active theme's folder. WordPress will fall back to a default theme.
  • Memory: Add define('WP_MEMORY_LIMIT', '256M'); to wp-config.php.
  • PHP version: Downgrade PHP in cPanel to the previous version while you update incompatible plugins.

Step 5: Restore From Backup (If Needed)

If the file-level fix doesn't work, restore from your most recent UpdraftPlus or hosting backup. Always restore the database and files together — partial restores can create version mismatches.

Prevention is cheaper: Every site I build has UpdraftPlus configured with daily automated backups. If you don't have backups, the recovery conversation gets much more expensive.

Elementor Pro vs Gutenberg: Which WordPress Builder Should You Use in 2026?

An honest comparison from a developer who uses both — including when I'd recommend one over the other to a client.

I'm not in the Elementor-or-nothing camp. I'm not in the Gutenberg-purist camp either. I use whichever tool fits the project. Here's how I decide.

When I Use Elementor Pro

  • Clients who need to edit visually. If the client isn't technical and wants to drag-and-drop content changes, Elementor is unbeatable. They can see exactly what they're changing in real time.
  • Marketing-heavy sites. Landing pages, funnels, A/B testing layouts, popup builders — Elementor Pro has these built in. With Gutenberg, you'd need multiple separate plugins.
  • Fast turnaround projects. When a client needs a full site in 5-7 days, Elementor templates and pre-built sections cut build time significantly.

When I Use Gutenberg + Custom CSS

  • Performance-critical sites. Gutenberg outputs cleaner, lighter HTML. No extra JavaScript libraries loading on every page. For sites where speed scores matter (Core Web Vitals), this makes a measurable difference.
  • Developer-managed sites. If I'm the one maintaining the site long-term, Gutenberg with ACF and custom blocks gives me more control with less bloat.
  • Simple content sites. Blogs, documentation sites, informational pages — these don't need a page builder. Gutenberg handles them natively.

The Honest Trade-offs

Elementor adds ~200-400KB of CSS/JS to every page load. On fast hosting with caching, users barely notice. On budget hosting without optimization, it's visible in performance audits. Gutenberg launches with near-zero overhead but requires more development time for complex layouts.

My default: If the client will be editing content themselves → Elementor Pro. If performance is the top priority or I'm managing the site → Gutenberg + ACF. There's no wrong answer — only wrong fits.

GoHighLevel for Beginners: How to Build Funnels, Affiliate Systems & Portals

Everything you need to understand before your first GHL build — including the parts no tutorial covers.

GoHighLevel is powerful. It's also overwhelming if you've never used it. This guide covers what I wish someone had told me before my first build.

What GoHighLevel Actually Is

It's an all-in-one platform that replaces your website builder, CRM, email marketing tool, funnel builder, appointment scheduler, and more. Think of it as ClickFunnels + Mailchimp + Calendly + a basic CRM, all in one dashboard.

Setting Up Your First Funnel

  1. Start with the goal. Lead capture? Product sale? Webinar registration? The funnel type determines your page structure.
  2. Use a template, then customize. GHL's funnel templates are solid starting points. Don't build from scratch unless you have a specific reason.
  3. Connect your domain. Always use a custom domain — not the default GHL subdomain. This is critical for trust and deliverability.
  4. Set up automation triggers. When someone fills out a form → tag them → email them → add to pipeline. This is where GHL's real power lives.

Building an Affiliate System

GHL has a built-in affiliate manager. You can create referral links, set commission structures (percentage or flat), and track conversions. The setup takes about 30 minutes, but the testing phase takes longer — always run test transactions before going live.

Client Portals

The membership/portal feature lets you create gated content areas. Clients log in, see their courses, resources, or deliverables. It's not as polished as Teachable or Kajabi, but it's good enough for most use cases — and it's included in the platform cost.

What tutorials won't tell you: GHL's email deliverability requires setup. You need to warm up your sending domain, configure DKIM/SPF, and start with low volume. Skip this step and your client's emails land in spam.

How to Migrate to GoHighLevel from WordPress or ClickFunnels (Complete Guide)

A practical migration guide for businesses switching from WordPress, ClickFunnels, or Kajabi to GHL.

Migrating an existing business to GoHighLevel isn't a one-click import. Here's how I approach it for clients.

What You Can Move

  • Contact lists: Export from your current CRM/email tool as CSV → import into GHL with field mapping. Clean duplicates first.
  • Email templates: Rebuild in GHL's email builder. Direct HTML imports work but often need tweaking for GHL's rendering engine.
  • Automation workflows: Map your existing automations on paper, then rebuild in GHL's workflow builder. This is the most time-consuming step.

What You Should Rebuild

  • Funnels and landing pages: ClickFunnels pages don't copy over. Rebuild them in GHL using templates — often faster than trying to replicate pixel-perfect.
  • Forms and surveys: Rebuild natively so they connect to GHL's pipeline and automation system.
  • Booking/calendar: Set up GHL's built-in calendar instead of trying to integrate Calendly. Fewer moving parts is always better.

What to Leave Behind

  • Complex WordPress sites: If your main website has 50+ pages with custom functionality, keep it on WordPress. Use GHL for funnels, CRM, and automation only.
  • Advanced e-commerce: GHL's e-commerce is basic. If you rely on WooCommerce features (subscriptions, variable products, tax rules), keep the store separate.
My migration timeline: Simple businesses (landing page + CRM + email) take 1-2 weeks. Complex setups (multiple funnels + affiliate system + membership portal + automations) take 3-5 weeks. Don't let anyone tell you it's a weekend project.

Website Migration Checklist: How to Switch Platforms Without Losing SEO Rankings

The checklist I run through on every platform migration to make sure rankings, redirects, and content survive the move.

I've migrated sites between WordPress, Squarespace, Wix, Webflow, and GoHighLevel. Every migration risks breaking SEO if you skip steps. Here's my exact checklist.

Before the Migration

  1. Crawl your current site. Use Screaming Frog or a similar tool to get a complete list of URLs, page titles, meta descriptions, and canonical tags. This is your source of truth.
  2. Export Google Search Console data. Download your top-performing pages, queries, and indexed URLs. You'll reference this to verify nothing drops after the move.
  3. Document all 301 redirects needed. If any URL structures will change, map old URLs → new URLs. Every single one.
  4. Backup everything. Full database, files, media, and content export. Store in two locations.

During the Migration

  1. Build the new site in staging. Never migrate directly on the live domain. Build, test, and verify on a staging URL first.
  2. Preserve URL structures. If you can keep the same URLs (/about, /services/design), do it. Changed URLs = redirect complexity.
  3. Implement 301 redirects. For every URL that changes, set up a permanent 301 redirect from the old path to the new one.
  4. Transfer meta data. Page titles, meta descriptions, and Open Graph tags should match or improve — never leave them blank.

After the Migration

  1. Submit the new sitemap to Google Search Console immediately.
  2. Test every redirect. Crawl the old URL list and verify each one resolves to the correct new page with a 301 status.
  3. Monitor Search Console for 30 days. Watch for Coverage errors, indexing drops, and 404 spikes.
  4. Check Core Web Vitals. The new platform may load faster or slower — verify and optimize if needed.
The #1 migration mistake: Forgetting to set up 301 redirects for old blog posts. Homepages and service pages get attention — but blogs often carry the most organic traffic. Redirect every single one.

Squarespace to WordPress Migration: Complete Step-by-Step Guide (2026)

From DNS to content export to plugin setup — a real migration walkthrough based on client projects.

I've moved several client sites from Squarespace to WordPress. It's not as simple as exporting and importing. Here's what actually happens.

Why Clients Switch

The most common reasons: they need WooCommerce for more advanced e-commerce, they want custom functionality Squarespace can't provide, or they've outgrown Squarespace's design limitations. Some need multi-author blogs with proper editorial workflows.

The Export Process

Squarespace exports content as an XML file — but it's limited. You get blog posts and basic pages. You don't get product data, form submissions, custom CSS, or media files in the export. Images stay hosted on Squarespace's CDN until you manually download and re-upload them.

DNS and Domain Transfer

If your domain is registered through Squarespace, you'll need to transfer it to a registrar like Namecheap or Cloudflare, or point the DNS to your new WordPress hosting. This step has a propagation period of up to 48 hours — plan for brief downtime.

What Gets Rebuilt

  • Design: Squarespace templates don't translate to WordPress. You're choosing a new theme and rebuilding the visual layout from scratch.
  • Forms: Squarespace forms don't export. Rebuild in a plugin like WPForms or Gravity Forms.
  • E-commerce: Product data (images, descriptions, variants, pricing) needs manual migration or a CSV import into WooCommerce.
  • SEO settings: Meta titles, descriptions, and custom URLs need to be manually re-entered using Rank Math or Yoast.
Timeline expectation: A simple Squarespace blog → WordPress migration takes 3-5 days. An e-commerce site with 100+ products takes 2-3 weeks. The content export is fast — the rebuild and QA are where the time goes.

5 Common Squarespace Errors That Are Actually DNS or Hosting Problems

SSL errors, broken domains, and blocked pages — here's how to diagnose the real cause before you blame the platform.

When a Squarespace site breaks, clients blame Squarespace. But in many cases, the issue is DNS, hosting configuration, or network-level. Here are five I've diagnosed and fixed.

1. "Site Not Secure" SSL Warning

Squarespace provides free SSL, but it only works if the DNS is configured correctly. If the domain is pointing through a third-party service (like Cloudflare proxy) without the right settings, the SSL certificate can't provision. Fix: Verify DNS records match Squarespace's requirements — CNAME for www, A records for root domain.

2. Domain Shows a Parking Page

The domain was pointed to Squarespace, but the client sees a "this domain is parked" page. This happens when the domain registrar still has default nameservers or A records. Fix: Update DNS at the registrar level. Clear your browser's DNS cache and wait for propagation.

3. Site Works on Some Networks, Not Others

This is almost always a Cisco Umbrella, corporate firewall, or ISP filtering issue. The site is live, but certain networks block it based on category filters. Fix: Check if the site loads on cellular data. If yes, the issue is the network — not Squarespace. Contact the network admin to whitelist the domain.

4. Email Not Working on the Domain

Squarespace doesn't provide email hosting. If the client sets up Google Workspace email (you@yourdomain.com) but doesn't configure MX records properly, emails bounce. Fix: Add Google Workspace MX records to the DNS zone. Don't forget to also add SPF, DKIM, and DMARC records for deliverability.

5. Redirect Loops After Custom Domain Setup

This happens when the client sets up both a www and non-www version of the domain but the DNS records conflict. Squarespace expects a specific configuration. Fix: Use their exact DNS setup guide — one A record pair for the root domain and one CNAME for www pointing to ext-cust.squarespace.com.

Before you call support: 80% of Squarespace issues I've been hired to fix were actually DNS misconfigurations. Always check DNS first.

5 AI Automation Workflows for Client Websites Using n8n & GoHighLevel

Practical automation workflows using Voiceflow, Botpress, Retell AI, and n8n that you can deploy on real client projects today.

AI automation isn't just for SaaS companies. I've added these workflows to client websites — local businesses, e-commerce stores, and service providers. Each one saves time, converts more leads, or reduces manual work.

1. AI-Powered Lead Qualification Chatbot

Tools: Botpress + GHL Pipeline

A chatbot embedded on the website asks qualifying questions (budget, timeline, project type). Based on answers, it either books a call via GHL's calendar or adds the lead to a nurture sequence. Unqualified leads get a polite resource page. This saves the client 5+ hours per week of pre-screening calls.

2. Automated Review Request Workflow

Tools: n8n + GHL + Google Business Profile

When a job is marked "complete" in GHL's pipeline, n8n triggers an automated workflow: wait 24 hours → send a personalized review request email → if no response after 3 days → send an SMS follow-up. This has increased review rates by 40%+ for service businesses.

3. AI Voice Agent for After-Hours Calls

Tools: Retell AI + GHL

When the business is closed, incoming calls are answered by an AI voice agent. It captures the caller's name, issue, and availability, then creates a contact in GHL with the call summary. The client wakes up to qualified leads instead of voicemails they'll never listen to.

4. Smart Content Personalization

Tools: Voiceflow + Website Integration

A Voiceflow assistant embedded on the site asks visitors what they're looking for. Based on responses, it dynamically recommends services, case studies, or pricing tiers. Instead of a generic homepage, each visitor gets a guided experience. Conversion rates increase because visitors see what's relevant to them immediately.

5. Automated Proposal Generation

Tools: n8n + OpenAI API + GHL

When a lead fills out a detailed intake form, n8n processes the responses, sends them to OpenAI to generate a draft proposal, and emails it to the client within minutes. The business owner reviews and sends — cutting proposal turnaround from 2 days to 2 hours.

The key insight: None of these replace human interaction. They replace the tedious parts — pre-screening, scheduling, follow-ups, and data entry. The business owner spends more time on high-value work and less on admin.
